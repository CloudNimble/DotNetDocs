# AOT Support for YAML Serialization

## Overview

Add Ahead-of-Time (AOT) compilation support to the YAML serialization pipeline in `CloudNimble.DotNetDocs.Core`.
YamlDotNet v12.3.0+ provides a source-generator-based static serialization context that eliminates reflection,
making it compatible with Native AOT and trimmed deployments.

Currently, `YamlRenderer` uses the reflection-based `SerializerBuilder`. This plan migrates it to use
`StaticSerializerBuilder` with a source-generated `StaticContext`, following the same pattern already
established in `Mintlify.Core` for System.Text.Json AOT support.

### Custom Type Converters
All 5 converters in `Renderers/YamlConverters/` are **write-only** (serialization only). `ReadYaml` throws
`NotImplementedException` on all of them. These converters handle Roslyn types that cannot be registered
with `[YamlSerializable]` since they're interfaces/enums from an external assembly.

## Implementation Plan

### Step 1: Add the Source Generator Package

**File**: `src/CloudNimble.DotNetDocs.Core/CloudNimble.DotNetDocs.Core.csproj`

Add a reference to the YamlDotNet source generator analyzer package:

```xml
<PackageReference Include="YamlDotNet.Analyzers.StaticGenerator" Version="16.*"
                  PrivateAssets="all" />
```

The `PrivateAssets="all"` ensures the analyzer doesn't flow to consuming projects.

### Step 3: Create the Static Context Class

**New file**: `src/CloudNimble.DotNetDocs.Core/Renderers/DotNetDocsYamlContext.cs`

```csharp
using YamlDotNet.Serialization;

namespace CloudNimble.DotNetDocs.Core.Renderers
{

    /// <summary>
    /// Source-generated static context for AOT-compatible YAML serialization
    /// of DotNetDocs model types.
    /// </summary>
    /// <remarks>
    /// This context enables the YamlDotNet source generator to produce reflection-free
    /// serialization code for all registered types, supporting Native AOT and trimmed deployments.
    /// Types from external assemblies (Microsoft.CodeAnalysis) are registered here since they
    /// cannot be decorated with <c>[YamlSerializable]</c> directly.
    /// </remarks>
    [YamlStaticContext]
    [YamlSerializable(typeof(Microsoft.CodeAnalysis.Accessibility))]
    [YamlSerializable(typeof(Microsoft.CodeAnalysis.SymbolKind))]
    [YamlSerializable(typeof(Microsoft.CodeAnalysis.TypeKind))]
    [YamlSerializable(typeof(Microsoft.CodeAnalysis.RefKind))]
    [YamlSerializable(typeof(Microsoft.CodeAnalysis.MethodKind))]
    public partial class DotNetDocsYamlContext : StaticContext
    {
    }

}
```

> **Note on Roslyn enums**: Even though these are handled by custom `IYamlTypeConverter` implementations,
> they must still be registered so the source generator knows about them. The custom converters will take
> precedence at runtime since they're registered on the builder.

### Step 5: Handle the Anonymous Type in `RenderTableOfContentsAsync`

**File**: `src/CloudNimble.DotNetDocs.Core/Renderers/YamlRenderer.cs` (lines 104-117)

The `RenderTableOfContentsAsync` method creates an anonymous type for the table of contents. Anonymous types
**cannot** be registered with `[YamlSerializable]` and will fail in AOT mode.

**Option A (Recommended)**: Create a dedicated TOC model class:

**New file**: `src/CloudNimble.DotNetDocs.Core/Renderers/YamlTableOfContents.cs`

```csharp
using System.Collections.Generic;
using YamlDotNet.Serialization;

namespace CloudNimble.DotNetDocs.Core.Renderers
{

    /// <summary>
    /// Represents a YAML table of contents for a documentation assembly.
    /// </summary>
    [YamlSerializable]
    public class YamlTableOfContents
    {
        /// <summary>
        /// Gets or sets the title of the documentation assembly.
        /// </summary>
        public string Title { get; set; } = string.Empty;

        /// <summary>
        /// Gets or sets the items in the table of contents.
        /// </summary>
        public List<YamlTocItem> Items { get; set; } = [];
    }

    /// <summary>
    /// Represents a single item in the YAML table of contents.
    /// </summary>
    [YamlSerializable]
    public class YamlTocItem
    {
        /// <summary>
        /// Gets or sets the display name of the namespace.
        /// </summary>
        public string Name { get; set; } = string.Empty;

        /// <summary>
        /// Gets or sets the relative file path to the namespace YAML file.
        /// </summary>
        public string Href { get; set; } = string.Empty;

        /// <summary>
        /// Gets or sets the types within this namespace.
        /// </summary>
        public List<YamlTocTypeEntry> Types { get; set; } = [];
    }

    /// <summary>
    /// Represents a type entry within a table of contents item.
    /// </summary>
    [YamlSerializable]
    public class YamlTocTypeEntry
    {
        /// <summary>
        /// Gets or sets the type name.
        /// </summary>
        public string Name { get; set; } = string.Empty;

        /// <summary>
        /// Gets or sets the kind of type.
        /// </summary>
        public string Kind { get; set; } = string.Empty;
    }

}
```

Then register these on the context class:
```csharp
[YamlStaticContext]
[YamlSerializable(typeof(YamlTableOfContents))]
[YamlSerializable(typeof(YamlTocItem))]
[YamlSerializable(typeof(YamlTocTypeEntry))]
// ... other registrations ...
public partial class DotNetDocsYamlContext : StaticContext { }
```

And update `RenderTableOfContentsAsync` to use the new types instead of the anonymous type.

**Option B**: Keep the anonymous type but serialize it with a separate reflection-based serializer.
This is not recommended as it defeats the purpose of AOT support.

**Files** (all in `src/CloudNimble.DotNetDocs.Core/Renderers/YamlConverters/`):
- `AccessibilityTypeConverter.cs`
- `RefKindTypeConverter.cs`

- `TypeKindTypeConverter.cs`

Add trimming/AOT suppression pragmas at the top of each file (matching the pattern used in `Mintlify.Core`):

```csharp

```

These converters use `Type` parameters which trigger AOT analyzer warnings, but they only do simple
type comparisons and string conversions — no actual reflection.

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Source generator doesn't handle abstract base class (`DocEntity`) | Low | YamlDotNet generates accessors for concrete types; abstract types are handled through inheritance. Test with a simple scenario first. |
| Custom `IYamlTypeConverter` conflicts with static context | Low | Custom converters registered on the builder take precedence over generated code. Existing behavior should be preserved. |
| `DocReference.TargetEntity` causes circular reference in generated code | Medium | Already handled by `DisableAliases()` and `WithMaximumRecursion(50)`. The `TargetEntity` property will serialize if populated, but in practice it's often null. If issues arise, add `[YamlIgnore]` to `TargetEntity`. |
| Baseline test output changes | Low | The static serializer should produce identical YAML output. If property ordering changes, regenerate baselines with `dotnet breakdance generate`. |
| `EnablePrivateConstructors()` unavailable on `StaticDeserializerBuilder` | Medium | Only affects tests. Either remove the call (source generator handles constructors) or keep the reflection-based deserializer in tests only. |
